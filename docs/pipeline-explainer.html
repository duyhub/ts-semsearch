How a Vietnamese query becomes ranked places

Users search maps by need, not by name — "quán cà phê yên tĩnh để làm việc" (a quiet café to work from). Keyword search can't read that. This walks the full engine end to end: every stage, the terminology, and the exact arithmetic that turns intent into an ordered, explained list. Its companion, pipeline-diagram.html, is the same story as one live diagram.

00The shape of it

A query flows through five stages. Nothing is a black box: retrieval is two classic methods fused, ranking is a weighted average of nine signals you can read off, and every reason shown to the user is checked against the data before it ships.

01Understand — text to structured intent

Vietnamese arrives messy: diacritics, typos, abbreviations (q1, hcm), English, even raw coordinates. Two steps tame it. Normalize does diacritic folding and abbreviation expansion — but only inside the search index. The displayed text always keeps its accents.

fold(s): NFD → strip combining marks → đ→d → lowercase → keep [a-z0-9/] → collapse whitespace # "Quán Cà Phê" → "quan ca phe" · "24/7" stays one token expand_query: q1 → "quan 1", hcm → "tp hcm", cf → "ca phe"

Parse then reads a structured intent off the folded text using the dataset's own closed vocabularies — the 12 categories and the fixed 10-attribute taxonomy. Anything it can't explain becomes residual: rare, place-naming leftovers turn into a subject constraint, and their mere presence blocks the category filter so a mis-parse can't over-prune.

02Retrieve — two views of the whole corpus, fused

BM25 is lexical: it rewards exact token overlap across a POI's name, category, address, attributes, tags and description. Strong when the query says the words. Blind to meaning. Dense retrieval fixes that: the bge-m3 embedding maps query and POI into the same 1024-d space, and cosine similarity measures closeness of meaning — so "nơi hẹn hò" finds romantic spots without the word appearing.

cosine(q, poi) = q · v_poi # both L2-normalized ⇒ one matrix-vector product

Their scores live on incompatible scales, so we fuse by rank, not score, with Reciprocal Rank Fusion. Each POI's fused score is normalized to [0,1] by a fixed maximum — never a per-query rescale — and that becomes the semantic signal downstream.

RRF(poi) = Σ_r 1 / (60 + rank_r(poi)) # r = BM25 list, dense list RRF_MAX = 2 / (60 + 1) # best case: rank 1 in both relevance = min(1, RRF(poi) / RRF_MAX)

Why this matters: because the semantic signal is the fused relevance, putting all ranking weight on it reproduces hybrid retrieval exactly. So the full ranker can only ever match or beat hybrid — a guarantee by construction, not luck.

03Rank — nine signals, one transparent score

Each signal maps a (query, POI) pair to [0,1], where 0.5 is neutral — constant across POIs, so a signal with no evidence can't change the order. Six map one-to-one to the sponsor's published ranking signals; category and price are our additions. The final score is a weighted average:

score(poi) = ( Σₖ wₖ · signalₖ(poi) ) / Σₖ wₖ # sort the full corpus by score

Weights are tuned by coordinate ascent on the tune split only (never the test split). Here is every signal, its calculation, and its tuned weight:

The two calculations worth spelling out

Rating uses Bayesian shrinkage so a 5.0 from three reviews doesn't beat a 4.6 from two thousand. The count v pulls the score toward the corpus mean C; a low prior m=30 keeps the narrow 3.8–4.7 rating band still varying instead of flattening out.

bayes = v/(v+m)·rating + m/(v+m)·C # m = 30, C = corpus mean rating rating = clamp01( (bayes − 3.5) / (5.0 − 3.5) )

Price is the only bidirectional signal — cheap intent scores level-1 POIs high, expensive intent inverts it.

norm = (price_level − 1) / (4 − 1) # 0 = cheapest … 1 = priciest price = 1 − norm if intent = cheap = norm if intent = expensive = 0.5 if no price intent or no price_level

04Constrain — honor what the query demanded

Re-ranking never deletes recall, but a query that says Quận 1 must not surface Quận 3. So hard constraints apply after ranking, as an AND-filter over location, subject and category. If the filter empties the list, the engine relaxes the most specific constraint first and retries — a valid query is never answered with nothing.

The subject filter has a subtlety worth knowing: it only fires when the dense retriever corroborates the term — some POI actually containing it sits in the dense top-10. That kills coincidental proper-name collisions (BM25 loves a rare token like nhất inside "Thống Nhất"; the embedding knows it isn't what the query meant).

When a location anchor resolves, the anchor gate floats nearby POIs to the top and drops far ones to the tail (recall preserved). It tries a tight radius, relaxes, and skips itself entirely rather than starve the result:

for radius in (30 km, 150 km): hits = POIs within radius of anchor if |hits| ≥ 3: return hits ++ rest # both keep score order otherwise: keep pure score order

05Serve — faithful reasons, exact contract

The top-k results are shaped into Tasco's published /v1/search contract exactly — an explicit judging ask — with stable poi: ids, WGS84 coordinates and diacritics preserved. The richer /v1/semantic-search adds each POI's per-signal breakdown, the parsed intent echo, and Vietnamese reasons.

Reasons are never asserted — each is generated from a verifiable fact (a matched attribute, distance to the anchor, the rating, opening hours) and then a validator re-checks the produced strings against the POI. Any attribute or number not backed by the data is rejected, so a hallucinated reason cannot ship. And if filters ever leave nothing, a backstop returns the most popular POIs, honestly stamped source="fallback" — the demo never shows an empty screen.

07How we know it works

The engine is measured, not asserted. On a committed, stratified tune/test split (the test set never touches code or weights), it reports real IR metrics: NDCG@5 0.96 and Recall@3 0.98 on the held-out test split, warm p95 ≈ 1 ms, and every one of 60 labeled queries plus adversarial inputs (emoji, empty, pure-English, unknown city, coordinates) returns 200 with at least one result. Numbers carry confidence intervals, and a test asserts no eval query is ever hardcoded to an answer. That measurement discipline is the edge — almost no hackathon team reports it.

Glossary — hover a dotted term
POI
Point of Interest — a specific place (café, restaurant, ATM, park…). The dataset is 111 POIs; the engine ranks them. Not "position" — a POI's lat/lon is one field it carries.
Diacritic folding
Stripping accents (cà phê → ca phe) for matching inside the index, while display keeps the accents.
Intent
The structured meaning parsed from a query: category, attributes, anchor, price, open-after, subject.
Attribute taxonomy
The dataset's fixed 10 attributes (wifi, yên tĩnh, phù hợp làm việc, lãng mạn…) used for constraint matching.
Residual
Query tokens the parser couldn't explain. Their presence blocks the category hard-filter (mis-parse guard).
Subject
A rare, POI-naming residual token (e.g. a dish) promoted to a hard filter — if dense corroborates it.
BM25
Classic lexical ranking by token overlap and rarity. Exact-match strong, meaning-blind.
Embedding
A vector capturing meaning. bge-m3 maps query + POI into the same 1024-d space, offline.
Cosine similarity
Angle-based closeness of two vectors, in [−1,1]. On L2-normalized vectors it's a single dot product.
Reciprocal Rank Fusion
Combines rankings by Σ 1/(60+rank) — fuses BM25 and dense without comparing raw scores.
Signal
One interpretable factor in [0,1] feeding the ranker (semantic, distance, rating…). Nine total.
Neutral (0.5)
A signal's no-evidence value. Constant across POIs, so it leaves the existing order unchanged.
Bayesian shrinkage
Pulling a rating toward the global mean by review count, so few-review scores don't dominate.
Bidirectional signal
One that can favor either extreme — price rewards cheap OR expensive depending on intent.
Hard constraint
A post-rank AND-filter (location/subject/category) that relaxes most-specific-first to stay non-empty.
Dense corroboration
Requiring the embedding retriever to also rank a term's POI highly before it may hard-filter.
Anchor
A resolved location (landmark, district centroid, POI) that powers distance and the anchor gate.
Weight tuning
Coordinate ascent picking signal weights on the tune split only — the test split never influences them.
Faithful explanation
A reason generated from POI facts and re-validated, so no attribute or number is fabricated.
