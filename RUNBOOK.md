# Plan A — Claude Code Runbook: build the whole thing in < 1 day

How to drive Claude Code so agents implement `semantic-ranking-implementation.md` end-to-end
while you review, steer, and prep the pitch.

## Operating model

- **You are the tech lead, Claude Code is the team.** You review gate results and make calls;
  agents write ~all code.
- **Everything is spec-driven.** The repo is seeded (before the event) with `PRD.md` (the
  requirements doc), `SPEC.md` (the implementation doc), `CLAUDE.md` (below), the xlsx, and
  the API PDF. Agents never need to invent requirements — only satisfy gates.
- **Phase-gated, test-first.** Each phase ends with a command whose output you (and the agent)
  can verify: pytest green + metric thresholds from SPEC §11. Commit at every passed gate.
- **Fresh context per phase.** Run `/clear` between phases; specs live in files so nothing is
  lost. This keeps agents fast and focused.
- **Parallelize with worktrees.** UI and API are independent of the ranking core once
  `search.py`'s interface is frozen (it's in SPEC §2) — run them as parallel sessions.

## CLAUDE.md to seed the build repo

```markdown
# Project: Tasco semantic search & ranking (AABW hackathon, 24h)
Read PRD.md (requirements — canonical) and SPEC.md (implementation design) before any work.

## Rules
- Python 3.11, uv. Run tests with `uv run pytest -q`. Type hints everywhere, no heavy frameworks.
- TDD for core logic: write/extend tests before implementation.
- NEVER hardcode any mapping from an eval query to POI ids anywhere in src/. The eval set is
  for measurement (tune split) only. Test split (see SPEC §6) must never influence code or weights.
- Vietnamese text: preserve diacritics in all outputs; folding only inside indexes.
- Every phase ends by running the gate command from SPEC §11 and reporting the metric table.
- Bedrock calls must have a working local fallback and a timeout. The demo can never depend on
  network success.
- Commit after each passed gate with message "gate GN passed: <numbers>".
- reports/ artifacts are generated by scripts, never hand-edited.
```

## Phase plan with prompts (~12h agent-clock; timeboxes generous)

Run phases sequentially in one main session (`/clear` between them). Phases marked ∥ run in
parallel worktree sessions.

### Phase 0 — Plan review (15 min, plan mode)
> Read PRD.md, SPEC.md, CLAUDE.md, and inspect data/raw/*.xlsx (all sheets). Produce a build
> plan for Phases 1–3 only: files, function signatures, test list. Flag any mismatch between
> PRD/SPEC and the actual data schema before we write code.

Approve the plan; fix schema surprises now, not at hour 6.

### Phase 1 — Data + eval harness (60 min) → *the ground truth everything iterates against*
> Implement SPEC §1 layout, §2 data contracts: data.py, scripts/ingest.py, eval.py,
> scripts/run_eval.py, plus tests/test_eval.py with hand-computed NDCG fixtures. Create the
> stratified 40/20 tune/test split (fixed seed) and commit it as data/derived/split.json.
> Gate: pytest green; run_eval.py --engine random prints a full metrics table (near-zero scores).

### Phase 2 — Normalization + BM25 baseline (60 min)
> Implement normalize.py per SPEC §3 (folding, abbreviation dict seeded from the taxonomy sheet)
> with tests, then retrieve.py BM25Index and a minimal search.py that returns BM25-ranked results.
> Gate G1: run_eval.py --engine bm25 --split tune → Recall@5 ≥ 0.55. If it fails, show me the
> 10 worst queries with their tokenizations and iterate on normalization until it passes.

### Phase 3 — Dense + hybrid (75 min)
> Implement embeddings.py (Bedrock cohere multilingual + bge-m3 fallback + disk cache, doc_text
> per SPEC §4), DenseIndex, rrf_fuse. Compare bedrock-vs-local on tune NDCG and pick the winner
> (record numbers in reports/embedding-choice.md).
> Gate G2: hybrid NDCG@5 > max(bm25, dense) on tune split. Print the 3-row comparison.

### Phase 4 — Ranker + tuning (90 min) → *the core ML deliverable*
> Implement geo.py (gazetteer from dataset + landmark list), rank.py signals per SPEC §6 table,
> LinearRanker with breakdown, tune.py coordinate ascent on tune split only. Then evaluate ONCE
> on test split.
> Gate G3: NDCG@5 ≥ 0.80 AND Recall@3 ≥ 0.75 on test. If short, analyze per-difficulty and
> per-query_category failures on the TUNE split only, improve signals (not memorization), rerun.
> Also run scripts/ablation.py and commit reports/ablation.md.

**Your job during Phase 4:** watch the failure analyses — this is where your ML judgment beats
the agent's. Typical fixes: anchor resolution misses, attribute canonicalization gaps, hard-filter
too aggressive.

### Phase 5 — Query parsing + explanations (75 min)
> Implement parse.py per SPEC §7 (rule parser + Bedrock structured-output parser + merge + cache
> + golden tests) and explain.py per SPEC §8 with the fact-faithfulness validator.
> Gate: golden parse tests green; run_eval.py with full pipeline does not regress G3; show 5
> example explanations for hard queries.

### Phase 6∥ — API (60 min, parallel worktree A)
> Implement api.py per SPEC §9 exactly matching the PlaceResult contract in docs/tasco_api.pdf
> (read it). tests/test_api_contract.py verifies shapes. Export openapi.json. Add
> scripts/bench_latency.py.
> Gate G4: contract tests green; warm p95 < 200ms over all 60 eval queries.

### Phase 7∥ — UI (2.5 h, parallel worktree B)
> Build ui/ per SPEC §10: search page with ranked cards (signal-breakdown bars, matched-attr
> badges, reasons), react-leaflet map with numbered pins, and the keyword-vs-semantic
> side-by-side toggle hitting /v1/search vs /v1/semantic-search. /metrics page renders
> reports/metrics.json + ablation. Vietnamese labels. Dark, clean, demo-legible at 1080p from
> 5 meters (big fonts).

Start Phase 7 as soon as Phase 4 passes (API shape is already fixed by SPEC); it runs while you
do Phases 5–6.

### Phase 8 — Integration + robustness (60 min, back to main session)
> Merge worktrees. Gate G5: robustness sweep — every one of the 60 eval queries plus this
> adversarial list (empty string, emoji, all-caps no-diacritics, 200-char rambling query, English
> query, pure address) returns HTTP 200 with ≥1 result and no exception. Fix everything found.
> Re-run all gates G1–G5 and commit.

### Phase 9 — Submission artifacts (45 min)
> Run make_samples.py → reports/samples.md (12 queries covering every query_category and
> difficulty). Write README.md per submission requirements (overview, setup, architecture,
> methodology, signals, models used). Deploy to App Runner (or give me the ngrok command).
> Then draft deck content as deck/outline.md following SPEC's demo script: problem → demo →
> architecture diagram → metrics/ablation → integration-readiness → roadmap.

### Phase 10 — You, not the agent (2 h)
Deck polish, 2× dress rehearsal against the live API, record backup video (screen capture of the
exact demo script), submit by 07:30.

## Contingency triggers

| Symptom | Action |
|---|---|
| Bedrock quota/latency issues | flip `EMBED_PROVIDER=local`; bge-m3 numbers were recorded in Phase 3, story unchanged |
| G3 stuck < 0.80 after 2 iterations | ship at best number with honest per-difficulty breakdown; judges reward measurement itself; pivot pitch to "hard-query analysis" |
| Next.js rabbit hole | cut to Streamlit fallback (SPEC §0) — 1 file, 90 min, still shows breakdown bars + map |
| Organizers release new/private queries at kickoff | they slot into eval.py as a new split; rerun tune.py; zero code changes |
| Wifi dies during demo | everything runs localhost incl. local embeddings + cached parses; OSM tiles pre-cached by rehearsal |
```
